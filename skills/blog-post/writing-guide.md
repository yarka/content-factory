# Writing Guide for Blog Posts

Стилистические правила для статей.

## Author Voice

**Тон:**
- Разговорный, как объясняешь другу
- Короткие предложения
- Без пафоса и "революционных прорывов"

**НЕ писать как:**
- Техписатель с формальным стилем
- Маркетолог с "revolutionary AI solution"

## Нарратив делегирования (для AI-статей)

**НЕ так:** "Сделай шаг 1, потом шаг 2..."
**А так:** "Я написал промпт → агент сделал → результат"

**Структура:**
1. Problem — что хотел решить
2. Промпт — цитата
3. Что агент сделал — конкретные шаги
4. Результат — работает, проверено

## Источник правды

**Статья пишется ТОЛЬКО на основе реальных данных:**
- Логи из Claude Code, Cursor и т.д.
- Git история проекта
- Реальные промпты и ответы

**НЕ выдумывать:**
- Диалоги с агентом
- Ошибки и их исправления
- Временные маркеры

Если нет логов — нет статьи. Фейковые истории читаются как фейк.

## Формула статьи

```
1. БОЛЬ — конкретная личная ситуация
2. ИДЕЯ — "а что если попросить агента"
3. ПЕРВАЯ ПОПЫТКА — что написал, что получил
4. ПРОБЛЕМА — что пошло не так
5. УТОЧНЕНИЕ — как переформулировал
6. РЕЗУЛЬТАТ — что работает, как использую
7. ИНСАЙТ — что понял про процесс
```

## Anti-patterns

❌ **File-centric** — перечисление модулей
```
"Архитектура: четыре модуля. sync.py, db.py..."
```

❌ **Code blocks** — примеры кода в статье
```
SELECT * FROM messages WHERE...
python search.py "запрос"
```

❌ **Tutorial style** — пошаговые инструкции
```
"Шаг 1: создай sync.py. Шаг 2: добавь rate limiting..."
```

## Patterns

✅ **Process-centric** — история делегирования
```
"Написал: сделай чтобы я мог искать по смыслу"
"Первая версия работала, но API забанил за спам"
"Уточнил: добавь random задержки между запросами"
```

✅ **Story style** — начало → проблема → решение
```
"За ночь синхронизировал 50 чатов — ни одного бана"
```

✅ **Разговорные промпты** — как объясняешь другу
```
"Сделай поиск по моим чатам. Хочу искать по смыслу, не по словам."
```

## Хуки

**15 секунд** — столько есть чтобы зацепить.

### Психологические триггеры

| Триггер | Пример |
|---------|--------|
| **Curiosity Gap** | "Думал, делаю всё правильно — пока не увидел эту метрику" |
| **Identity** | "Если твои первые проекты провалились — добро пожаловать" |
| **Tension** | "Удвоил продуктивность — и стало хуже" |

### Типы лидов

| Тип | Пример |
|-----|--------|
| **Zinger** | "Твой CLAUDE.md — мусор" |
| **First-Person** | "Попросил агента настроить — он сломал всё" |
| **Question** | "Почему агент тормозит после 50 сообщений?" |
| **Scene** | "Открыл терминал. Написал промпт. Три часа спустя — готовый продукт" |

### Формула intro (SPY)

**S**hort — короткий удар (1-2 предложения)
**P**ain — боль читателя
**Y**ay — неожиданный поворот

### Антипаттерны

❌ "В современном мире AI..."
❌ "Многие разработчики сталкиваются..."
❌ "Сегодня мы рассмотрим..."

✅ Конкретная ситуация или проблема
✅ Личное "я" вместо "мы"
✅ Максимум 30 слов на хук

## Ритм текста

**Длина предложений:**
- Больше 23 слов — разбивай
- Энергичный текст: до 11 слов

**Рваный ритм:**
- Чередуй: длинное → короткое → среднее
- Абзац из одного предложения = акцент

**Проверка:** Читай вслух. Споткнулся — переписывай.

## Языковые правила

Пиши на русском. Избегай англицизмов.

**Справочник:** `ai-terms-ru.md`

Ключевые замены:
- claim → утверждение
- generic → шаблонный
- insight → наблюдение

Оставлять на английском:
- Имена продуктов: Claude Code, GPT
- Аббревиатуры: LLM, RAG, API
- Устоявшиеся: промпт, токен, агент

## Checklist перед публикацией

- [ ] **Основано на реальных данных** (логи, git)?
- [ ] Статья начинается с личной боли, не технологии?
- [ ] Промпты читаются как разговор?
- [ ] Есть итерации (попытка → проблема → решение)?
- [ ] Нет перечисления файлов/модулей?
- [ ] Есть временные маркеры ("за вечер", "через час")?
- [ ] Финал про результат для пользователя, не про код?
